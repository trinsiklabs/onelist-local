Onelist Free Tier: Self-Hosted

Status: Planning Updated: 2026-01-28

1. Vision & Goal

Onelist offers a completely free tier that requires no monthly payment and no cloud storage dependency. Users run their own Onelist infrastructure and sync between their endpoints using peer-to-peer protocols.

Goals:
  • Lower barrier to entry for technical users
  • Attract privacy-conscious users who prefer self-hosting
  • Build user base for community features (forums)
  • Entry into ecosystem (account, identity, email communications)
  • Natural upgrade path to paid tiers

    Free tier user posts in forum
         │
         ▼
Post stored in ONE of:
├── Onelist corporate forum (general discussions)
├── Another user's Onelist-hosted forum (comments on their content)
         │
         ▼
Post is:
├── Tied to their account as originator
├── Stored UNENCRYPTED (forums are public/readable)
├── NOT synced to their personal Onelist (no cloud connection)
         │
         ▼
If user upgrades to Cloud Sync later:
└── Historical posts can be copied to their Onelist

    6.2 Forum Hosting Model

    Users with Cloud Sync or VM tier can run their own forums on their Onelist-hosted sites. Other users (including free tier) can participate:

  • Using their Onelist account
  • Or anonymously

    • Forum content is stored on the forum host's Onelist, creating a federated/decentralized forum system.

    9. P2P Sync Technical Specification

    9.1 Change Tracking

    Each syncable record includes:

    -- Added to all synced tables
sync_version bigint NOT NULL DEFAULT 0;
sync_clock jsonb NOT NULL DEFAULT '{}';      -- Vector clock
sync_deleted boolean NOT NULL DEFAULT false;  -- Soft delete for sync
sync_origin_device_id uuid;                   -- Which device created this
    Vector Clock Format: 
    {"device_a_id": 42, "device_b_id": 37, "device_c_id": 15}

    9.2 Database Schema

    -- Device registry (synced between devices)
CREATE TABLE devices (
  id uuid PRIMARY KEY,
  account_id uuid NOT NULL REFERENCES users(id),
  name text NOT NULL,
  public_key bytea NOT NULL,           -- Ed25519 public key
  device_type text,                     -- desktop, server
  os_info text,
  last_seen_at timestamptz,
  registered_at timestamptz NOT NULL,
  sync_deleted boolean DEFAULT false
);

    -- Sync state per peer (local to each device)
CREATE TABLE sync_peers (
  id uuid PRIMARY KEY,
  local_device_id uuid NOT NULL REFERENCES devices(id),
  peer_device_id uuid NOT NULL REFERENCES devices(id),
  last_sync_at timestamptz,
  last_sync_version bigint DEFAULT 0,
  connection_hints jsonb,               -- [{type, ip, port}, ...]
  status text DEFAULT 'unknown',
  UNIQUE(local_device_id, peer_device_id)
);
    

    -- Pending sync operations (local to each device)
CREATE TABLE sync_outbox (
  id uuid PRIMARY KEY,
  table_name text NOT NULL,
  record_id uuid NOT NULL,
  operation text NOT NULL,              -- insert, update, delete
  payload jsonb NOT NULL,
  sync_version bigint NOT NULL,
  created_at timestamptz DEFAULT now(),
  synced_to jsonb DEFAULT '{}'          -- {peer_id: timestamp, ...}
);
    

    -- Unresolved conflicts (local to each device)
CREATE TABLE sync_conflicts (
  id uuid PRIMARY KEY,
  table_name text NOT NULL,
  record_id uuid NOT NULL,
  local_version jsonb NOT NULL,
  remote_version jsonb NOT NULL,
  local_clock jsonb NOT NULL,
  remote_clock jsonb NOT NULL,
  conflict_type text NOT NULL,
  resolved boolean DEFAULT false,
  resolution text,
  resolved_at timestamptz,
  created_at timestamptz DEFAULT now()
);
    

    -- Asset mirror status (local to each device)
CREATE TABLE asset_mirrors (
  id uuid PRIMARY KEY,
  asset_id uuid NOT NULL REFERENCES assets(id),
  location text NOT NULL,               -- 'local', 'cloud:r2', 'peer:device_id'
  status text NOT NULL,                 -- pending, syncing, synced, failed
  storage_path text,
  checksum text,
  synced_at timestamptz,
  error_message text,
  retry_count integer DEFAULT 0,
  UNIQUE(asset_id, location)
);

    9.3 Sync Protocol Flow

    INITIATOR (Device A)                         RESPONDER (Device B)
─────────────────────                        ─────────────────────

    1. HELLO
   {device_id, account_id,
    protocol_version, auth_challenge}
                                    ────────►
                                              2. Verify account ownership
                                                 Sign challenge with device key
                                    ◄────────
   {auth_response, peer_sync_version}
    

    3. SYNC REQUEST
   {since_version: 42,
    tables: ["entries", "tags", ...]}
                                    ────────►
                                              4. Gather changes since version 42
                                    ◄────────
   {changes: [...],
    current_version: 67,
    has_more: false}
    

    5. Apply changes locally
   Detect conflicts via vector clocks
    

    6. SYNC RESPONSE (our changes)
   {changes: [...],
    current_version: 58}
                                    ────────►
                                              7. Apply changes
                                                 Detect conflicts
    

    8. CONFLICT RESOLUTION (if any)
   {conflicts: [...],
    resolutions: [...]}
                                    ◄───────►
                                              Bidirectional until resolved
    

    9. SYNC COMPLETE
   {final_version: 70}
                                    ────────►
                                    ◄────────
   {ack}

    9.4 Conflict Resolution

    Conflict TypeStrategy
    Concurrent edit (trivial)Last-Write-Wins (LWW)
    Concurrent edit (significant)Flag for user resolution
    Delete vs EditPrefer edit unless delete is newer
    Tag changesMerge sets

    9.5 Security

    Device Authentication:
  • Each device has Ed25519 keypair
  • Public key registered with account
  • Challenge-response authentication between peers
    • Transport Encryption:
  • X25519 key exchange for session key
  • ChaCha20-Poly1305 authenticated encryption
  • All P2P traffic encrypted
    • Key Exchange Flow: 
    1. Devices exchange ephemeral X25519 public keys
2. Derive shared secret via ECDH
3. Derive session keys using HKDF
4. All subsequent frames encrypted with session key

    9.6 Asset Sync

    Storage Options (per device):
  • local_only - Assets on local filesystem only
  • cloud_only - Assets in BYOB bucket only
  • local_and_cloud - Mirrored to both
    • Asset Transfer:
  • Chunked streaming (256 KB chunks)
  • Checksums for integrity verification
  • Resume capability for interrupted transfers
    • Sync Priority: 
    1. Metadata always syncs (small, fast)
2. Full assets sync based on device config:
   
  • If peer has file + I want local → P2P transfer
    • 
   
  • If peer has no file + cloud has file → Fetch from cloud
    • 
   
  • If I don't want local → Just metadata (fetch on demand)

    10. WAN Discovery Service (Post-MVP)

    See: Corporate Services Plan - Section 4: Discovery Service for full implementation details including API, schema, and NAT traversal.
    Summary for Free Tier:
  • Discovery service is operated by Onelist (part of Corporate Services)
  • Enables P2P sync across the internet (WAN), not just LAN
  • Coordinates NAT hole punching between devices
  • Does NOT relay traffic - only coordinates device finding
  • If hole punch fails, users must use Tailscale/VPN or upgrade to Cloud Sync for relay access

    VM Tier Disclosure:
    Your Onelist VM runs on infrastructure managed by Onelist. While the filesystem is encrypted at rest, your content is decrypted while the server is running to enable full functionality including AI agents, search, and API access. This means Onelist technically has access to your content. If you require zero-knowledge privacy, use Cloud Sync tier with your own endpoint.
