Purpose: Virtual life manager, coach, assistant, and cheerleader that maintains a comprehensive view of everything the user is trying to accomplish across all domains of life
Methodology Foundation: David Allen's Getting Things Done (GTD)

1. Vision & Philosophy

1.1 Core Concept

The Life Steward is an intelligent agent that sees the user's entire life as a manageable system. Unlike other agents that process individual entries reactively, the Life Steward maintains and evolves a hierarchical life management structure and proactively helps the user organize, prioritize, and accomplish their goals.

The Life Steward is:
  • A life operations manager - keeps everything organized and nothing falls through the cracks
  • A coach - helps identify what matters and keeps user focused on goals
  • A virtual assistant - handles the cognitive overhead of life organization
  • A cheerleader - celebrates wins and provides encouragement
  • An intelligent filing system - automatically routes incoming entries to the right place
  • A timeline analyst - estimates durations, tracks velocity, and identifies critical paths

    • 1.2 GTD Foundation

    The Life Steward is deeply informed by David Allen's Getting Things Done methodology. GTD's core insight is that mental clarity comes from having a trusted external system that captures everything, enabling a "mind like water" state where you respond appropriately to inputs without anxiety about forgotten commitments.

    GTD's Five Steps (Mapped to Life Steward):
    GTD StepDescriptionLife Steward Implementation
    CaptureGet everything out of your headInbox for all incoming entries; integrations with OpenClaw, Feeder, etc.
    ClarifyProcess: Is it actionable? What's the next action?Entry processor determines type, identifies next actions
    OrganizePut things where they belongDomain/Scope/Project hierarchy + GTD lists
    ReflectRegular reviews to maintain trustDaily, Weekly, Monthly reviews (Weekly Review is cornerstone)
    EngageDo the work with confidenceContext-based views, today lists, focus modes
    GTD's Two-Minute Rule: If processing reveals an action that takes less than two minutes, do it immediately rather than tracking it. The Life Steward will flag "quick wins" during processing. GTD's Natural Planning Model: For projects, the Life Steward guides users through: 1. Purpose - Why does this project matter? 2. Vision - What does "done" look like? 3. Brainstorm - What ideas, resources, constraints exist? 4. Organize - What's the structure and sequence? 5. Next Actions - What's the very next physical action?

    1.5 Design Philosophy

    1. User remains in control - Agent recommends, user accepts/rejects 2. Flexibility over rigidity - Structure adapts to user's life, not vice versa 3. Progressive disclosure - Start simple, complexity grows as needed 4. Everything is promotable/demotable - Any entry can become a project, any project can become a domain 5. Intelligent defaults - Standard buckets work for most people, custom buckets for power users 6. Next Actions, not vague tasks - Every task should be a concrete, physical next step 7. Weekly Review is sacred - The cornerstone practice that maintains system trust 8. Bottom-up control before top-down planning - Master daily chaos before strategizing

  • Acts as catch-all for entries that don't fit elsewhere
  • Cannot be deleted, only renamed

    5. Timeline Estimation & Critical Path Analysis

    5.1 Philosophy: Practical Timeline Estimation

    GTD intentionally de-emphasizes time estimation - Allen argues that most knowledge work is difficult to estimate and that "hard landscape" (calendar-bound) items should be few. However, for project planning and life management, timeline visibility has value when approached pragmatically.

    Life Steward's Approach:
  • Estimates are ranges with confidence levels, not false precision
  • Estimates are dynamic - continuously updated based on actual progress
  • User velocity (historical completion rate) informs all estimates
  • The system distinguishes effort (how long the work takes) from duration (calendar time including delays)
  • Estimates are optional - users can use the system without ever estimating
    • When Timeline Estimation is Valuable:
    ScenarioValue
    Hard deadlines (taxes due, event date)Critical - must work backward
    Coordinating with othersHigh - need to communicate expectations
    Large projects with dependenciesHigh - identify bottlenecks early
    Personal goals without deadlinesLow - focus on next actions, not dates
    Creative/exploratory workLow - estimates often counterproductive

    5.5 Critical Path Analysis

    The critical path is the longest sequence of dependent tasks that determines the minimum project duration. Tasks on the critical path have zero "slack" - any delay directly delays the project.

    5.7 Practicality Assessment: When Estimation Helps vs. Hurts

    Timeline Estimation Works Well For:
    ScenarioWhy It Helps
    Projects with hard deadlinesMust work backward from deadline
    Multi-person coordinationNeed shared expectations
    Sequential dependenciesDelays cascade predictably
    Familiar, repeatable workHistorical data is reliable
    External commitmentsOthers depend on your timeline

    ScenarioWhy It Hurts
    Creative/exploratory workInherently unpredictable
    First-time novel workNo historical basis for estimates
    Research with unknown scopeCan't estimate what you don't know
    Personal goals without deadlinesAdds pressure without benefit
    Rapidly changing requirementsEstimates obsolete quickly

    6. User Interaction Modes

    6.1 Review Sessions

    GTD emphasizes that the Weekly Review is the cornerstone habit that keeps the system trusted. Without regular reviews, any productivity system becomes stale and loses the user's trust.

    Daily Review (5-10 min) - "Get Clear":
  • Review calendar for today and tomorrow
  • Review Next Actions list - what's realistic today?
  • Check Waiting For list - any follow-ups needed?
  • Process inbox items that arrived overnight
  • Quick wins: anything < 2 minutes, do now
    • Weekly Review (60-90 min) - "The Cornerstone":

    Per GTD, the Weekly Review has three phases:

    ┌─────────────────────────────────────────────────────────────────────────────┐
│                          GTD WEEKLY REVIEW                                   │
│                                                                              │
│  PHASE 1: GET CLEAR (Empty all inboxes)                                     │
│  ─────────────────────────────────────────────────────────────────────────  │
│    □ Process all entries in Inbox                                           │
│    □ Empty email/physical inbox to zero (or reference)                      │
│    □ Review loose notes, voicemails, meeting notes                          │
│    □ Empty your head - capture anything still floating around               │
│                                                                              │
│  PHASE 2: GET CURRENT (Review all lists)                                    │
│  ─────────────────────────────────────────────────────────────────────────  │
│    □ Review Next Actions lists - still relevant? Complete? Re-context?      │
│    □ Review Waiting For list - need to follow up on any?                    │
│    □ Review Project list - each has a next action? Any complete?            │
│    □ Review Someday/Maybe - anything to activate? Anything to delete?       │
│    □ Review calendar (past week: capture follow-ups)                        │
│    □ Review calendar (upcoming: prepare, add actions)                       │
│                                                                              │
│  PHASE 3: GET CREATIVE (Reflect and plan)                                   │
│  ─────────────────────────────────────────────────────────────────────────  │
│    □ Review Goals - progress this week? Actions aligned?                    │
│    □ Review Areas of Focus (Domains/Scopes) - anything neglected?           │
│    □ Consider: any new projects emerging?                                   │
│    □ Consider: any projects to delegate, defer, or drop?                    │
│    □ Timeline review: any projects at risk? Dependencies clear?             │
│    □ Accept/reject Life Steward recommendations                             │
│                                                                              │
│  ✓ WEEKLY REVIEW COMPLETE                                                   │
│    "Mind like water" - ready for the week ahead                             │
└─────────────────────────────────────────────────────────────────────────────┘
    Life Steward's Weekly Review Enhancement:

    The agent prepares a Weekly Review briefing:

  • Pre-sorted inbox items with suggested classifications
  • Stale items flagged (no activity in 14+ days)
  • Waiting For items past expected response date
  • Projects with no next actions defined
  • Timeline risks and critical path alerts
  • Domain balance analysis (time spent vs. goals)
  • Celebration: completed projects and progress highlights

    • Monthly Review (30-60 min) - "Horizon Check":
  • Review Horizon 2: Areas of Focus (Domains/Scopes) - still accurate?
  • Review Horizon 3: Goals (1-2 year) - on track?
  • Habit effectiveness review - what's working?
  • Structural changes - domains/scopes to add/archive/merge?
  • Velocity analysis - are estimates calibrated?
  • Life balance check - any areas severely neglected?
    • Quarterly Review (90-120 min) - "Strategic Altitude":
  • Review Horizon 4: Vision (3-5 year) - still inspiring?
  • Review Horizon 5: Purpose & Principles - still true?
  • Major project planning for upcoming quarter
  • Cross-project dependencies and resource allocation
  • Archive completed/abandoned projects
  • Celebrate major accomplishments

    7.6.5 Conversation Design Principles

    Minimize Cognitive Load:
  • Agent drives the conversation; user just responds
  • Default suggestions reduce decision fatigue ("Should I file this under X?")
  • Quick responses accepted: "yes", "no", "done", "skip", "next"
  • Agent remembers context within session ("same as last one")
    • Graceful Corrections:
  • "Actually..." triggers edit mode
  • "Wait, go back" returns to previous item
  • "No, I meant..." allows re-statement
  • Confirmation before destructive actions
    • Adaptive Pacing:
  • Detects when user needs time to think (silence handling)
  • Speeds up or slows down based on response patterns
  • "Take your time" mode for thoughtful reviews
  • "Quick mode" for rapid triage
    • Natural Language Understanding:
    User SaysAgent Understands
    "Done" / "Finished" / "I did that"Mark complete
    "Skip" / "Next" / "Move on"Keep, proceed to next
    "Nah" / "Delete it" / "Never mind"Remove item
    "Later" / "Someday" / "Not now"Move to Someday/Maybe
    "Move to [project]"Refile entry
    "Add a note..."Create reference entry
    "That reminds me..."Capture new item, then return
    "What's next?"Proceed in flow
    "Where are we?"Summarize current state
    "How much longer?"Estimate remaining time

    11. Open Questions

    10.1 Design Questions

  • [ ] Q1: How deeply should Life Steward integrate with existing tag system? Should domains/scopes/projects be tags, or separate entities?
  • Leaning: Separate entities stored as special entry types, but can auto-generate tags for compatibility
  • [ ] Q2: Should there be a limit on hierarchy depth? (Domain → Scope → Project → ???)
  • Leaning: Three levels is enough per GTD (Areas → Projects → Actions); deeper nesting creates complexity
  • [x] Q3: How to handle entries that genuinely belong in multiple places?
  • Decision: Primary location + links to related locations; projects can be "visible in" multiple domains
  • [ ] Q4: What happens to entries when a project is archived?
  • Leaning: Entries stay with archived project, searchable but hidden from active views
  • [ ] Q5: Should there be templates for common domain types (Business, Creative Project, etc.)?
  • Leaning: Yes, with suggested scopes, buckets, and Natural Planning prompts
  • [ ] Q6: How strictly to enforce GTD vocabulary vs. user's preferred terms?
  • Leaning: Default to GTD terms with option to customize; help text explains mapping

    • 10.2 Timeline & Estimation Questions

  • [ ] Q7: Should effort estimates be required or optional?
  • Leaning: Always optional; agent recommends estimation mode based on project type
  • [ ] Q8: How to handle estimation for creative/exploratory work?
  • Leaning: Recommend minimal estimation mode; track output metrics (word count, commits) instead
  • [ ] Q9: What's the right confidence interval representation for non-technical users?
  • Leaning: Simple "Optimistic / Expected / Pessimistic" with visual range bar
  • [ ] Q10: Should velocity tracking be automatic or opt-in?
  • Leaning: Opt-in with prompt when user starts tracking actuals; explain benefit
  • [ ] Q11: How to handle projects with many unknowns ("research" phase)?
  • Leaning: Support milestone-based estimation ("Research done by X, then re-estimate")

    • 10.3 Critical Path Questions

  • [ ] Q12: How to visualize cross-domain dependencies without overwhelming?
  • Leaning: Only show when relevant (project view, timeline risk alerts); don't clutter day-to-day
  • [ ] Q13: Should resource contention be automatic (based on time) or manual?
  • Leaning: Start manual ("I can spend 10h/week on this"); optionally infer from calendar
  • [ ] Q14: How to handle soft vs. hard dependencies?
  • Leaning: Both types; soft dependencies don't affect critical path calculation

    • 10.4 Technical Questions

  • [ ] Q15: Real-time entry processing vs. batch processing?
  • Leaning: Near real-time with batched notifications
  • [ ] Q16: How to handle classification learning/improvement over time?
  • Leaning: Track user corrections, adjust confidence accordingly
  • [ ] Q17: Where does Life Steward state live? (its own DB? Onelist entries?)
  • Leaning: Everything in Onelist as entries (domains, scopes, projects, timeline metadata all in entries)
  • [ ] Q18: How to efficiently compute critical path for large project graphs?
  • Leaning: Compute on-demand and cache; invalidate when dependencies change

    • 10.5 UX Questions

  • [ ] Q19: How prominent should Life Steward suggestions be? Interrupting vs. passive?
  • Leaning: Configurable; default to batched/passive
  • [ ] Q20: How to onboard users to the domain/scope/project concept without overwhelming?
  • Leaning: Start with just Personal domain + inbox; GTD "mind sweep" capture exercise; suggest structure as entries accumulate
  • [ ] Q21: Should Weekly Review be enforced (nagging) or gentle (available)?
  • Leaning: Gentle by default with configurable commitment mode for users who want accountability

    • 10.6 Voice Feature Questions

  • [ ] Q22: Local STT (Whisper) vs cloud STT - how to handle the tradeoff?
  • Leaning: Local by default on capable devices; cloud as opt-in for better accuracy or less powerful devices
  • [ ] Q23: Should voice sessions be recorded/transcribed by default?
  • Leaning: Transcript saved (for user reference); audio discarded immediately (privacy)
  • [ ] Q24: How to handle voice recognition errors gracefully?
  • Leaning: Show transcript in real-time; "I heard X, is that right?" for low-confidence; easy "that's not right" correction
  • [ ] Q25: What's the minimum viable voice feature for initial release?
  • Leaning: Weekly Review flow with basic commands (done/skip/keep); capture new items; UI follows along
  • [ ] Q26: How to handle multi-person households (voice identification)?
  • Leaning: Session start requires visual confirmation or PIN; don't rely on voice ID initially
  • [ ] Q27: Should voice work when app is in background/locked?
  • Leaning: Requires app open initially; background listening is future enhancement with clear privacy controls

    • 10.7 Multi-Agent Orchestration Questions

  • [ ] Q28: How to handle agent failures mid-pipeline (Researcher fails, Writer waiting)?
  • Leaning: Retry with backoff; notify user after N failures; allow manual intervention
  • [ ] Q29: Should research/writing agents run on user's LLM keys or Onelist's?
  • Leaning: User's keys by default (cost control); Onelist keys as premium option
  • [ ] Q30: How to handle conflicting content updates (two sources update same chapter)?
  • Leaning: Queue for user triage; show both sources; let user decide merge strategy
  • [ ] Q31: What's the right granularity for human approval gates?
  • Leaning: Configurable per project; default to review research + review drafts
  • [ ] Q32: How long should research/writing tasks wait before timing out?
  • Leaning: Configurable; default 24h for research, 48h for writing; notify on delay
  • [ ] Q33: Should living documents support multiple simultaneous editions in progress?
  • Leaning: Initially no (one edition at a time); future enhancement for parallel tracks
  • [ ] Q34: How to handle external publishing failures (Xamlr down, PDF generation fails)?
  • Leaning: Retry; keep edition ready; notify user; don't block future editions
