Established Design Conventions

These conventions have been reviewed and approved across all agent UI plans:

1. List View as Universal Pattern

  • Always use "list view" - never "table view" or "data table"
  • List view is the primary UI pattern everywhere in Onelist (fits the app name)
  • Even chat interfaces use list view under the hood
  • Cards view (Pinterest-style) is an alternative toggle, not the default

    • 2. Confidence/Quality Display

  • Always use percentage bars with text: ‚ñà‚ñà‚ñà‚ñà‚ñà‚ñà‚ñà‚ñà‚ñë‚ñë 87%
  • Color coding:
  • Green: 85-100% (high confidence)
  • Yellow: 60-84% (review suggested)
  • Red: <60% (review recommended)
  • Never use dot notation (‚óè‚óè‚óè‚óã) for confidence

    • 3. Person Entity Integration

  • Speakers, owners, assignees link to Person entities when known
  • Unknown speakers show as "Speaker 1", "Speaker 2", etc.
  • Click to identify/link via inline dropdown (not dialog)
  • Voiceprints remembered for future recognition

    • 4. Inline Editing Pattern

  • Click-to-edit for names, text, assignments
  • Inline dropdowns for selection (not modal dialogs)
  • Auto-save changes

    • 5. Expandable Row Details

  • Quality issues, context, actions shown in expandable rows
  • Nested under parent row in list view

    • 6. Classification Confidence Thresholds

  • >85%: Auto-file, notify user
  • 60-85%: Suggest, await confirmation
  • <60%: Present options, user decides
