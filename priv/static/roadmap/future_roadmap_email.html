Onelist Feature Branch Exploration: Email Integration

Status: Conceptual Exploration (Post-v2.0 Roadmap Completion) Date: 2025-05-06

1. Vision & Goal

This document explores a potential future direction for Onelist, initiated after the successful implementation of the core v2.0 roadmap (including advanced AI features and daemons). The vision is to extend Onelist's role as the central "augmented memory" by integrating email directly into its ecosystem, transforming it into a full-featured email client and repository.

Goal: Address the challenge of scattered information across multiple inboxes by treating emails as first-class Onelist Entries. This allows emails to be stored permanently, searched universally alongside other knowledge, linked contextually, and acted upon by Onelist's AI capabilities and automated daemons, ultimately consolidating a major communication channel into the user's unified knowledge base.

2. Core Concepts & Features

Integrating email functionality would involve several key capabilities:

A. Full Email Client Functionality: Multi-Account Connectivity: Support connecting multiple external email accounts via standard protocols (IMAP for fetching, SMTP for sending) and potentially specific vendor APIs (e.g., Gmail, Outlook) using secure authentication (OAuth2 where applicable). Email Operations: Fetching, receiving, reading, composing, replying, and forwarding emails directly within the Onelist interface. Storage: Securely storing fetched and sent emails within the Onelist system. Folder/Label Management: Synchronizing email folders/labels and potentially mapping them to/from Onelist tags. Attachment Handling: Storing and accessing email attachments. B. Emails as Native Onelist Entries: Data Model: Each email becomes an entry with entry_type='email'. Headers (From, To, Cc, Subject, Message-ID, etc.) stored efficiently (potentially in a dedicated email_details side-table for threading/querying). Email bodies stored as representations (e.g., 'text/plain', 'text/html', 'original_eml'). Attachments stored as linked assets. entries.content_created_at stores the email's timestamp. C. Deep Integration with Onelist Ecosystem (Leveraging v2.0 Capabilities): Universal Search: Emails indexed by the Onelist Searcher (FTS and semantic embeddings) for unified search across all knowledge types. Contextual Linking: Ability to link emails to/from other Onelist entries (notes, tasks, projects, chat logs) using entry_links for rich context. Daemon Interaction: Emails become inputs for automated processing by daemons: Planner: Derive tasks, flag follow-ups. Reader: Summarize emails/threads. Researcher: Find related Onelist content based on email topics. Librarian: Suggest tags based on email content/context. AI Interaction: Contextual AI Chat Pane: Operate on email entries (summarize, draft replies, extract information, suggest linked tasks/notes). "Onelist AI" (Global Assistant): Query across all emails, draft new emails, manage email-derived tasks via natural language. D. Enhanced Email Workflows: Task Derivation: Streamlined features to create Onelist Tasks (e.g., entry_type='task') directly from email content, automatically linking the task back to the source email. Follow-up Management: System for tracking emails requiring follow-up actions. Derived Task Tracking: Visibility into the status of tasks generated from specific emails. Actionable Content Extraction: Simplified workflows for extracting specific information (like links for wishlists, recipe details, meeting times) from emails and creating relevant linked Onelist entries.

3. Impact on Existing Onelist System (Post-v2.0)

Implementing this requires significant additions and modifications:

Data Model: Introduction of entry_type='email' and potentially a dedicated email_details side-table. Heavy reliance on representations, assets, and especially entry_links. New Backend Service(s): A dedicated, robust service is needed for handling email protocol connections (IMAP/SMTP/APIs), synchronization, parsing, and state management. Daemon & AI Enhancements: Existing daemons (Planner, Reader, Searcher, etc.) and AI components need updates to process the 'email' entry type and associated metadata/workflows. API Expansion: New API endpoints for email-specific actions and management. Major Frontend UI Development: Requires building a comprehensive email interface within Onelist (account management, list views, read/compose views, folder/tag integration). Security Infrastructure: Enhanced handling of user credentials (OAuth tokens) for external email services. Safe rendering of HTML email content. Spam filtering considerations.

4. Potential MVP for Email Integration Branch (Post-v2.0)

A phased approach would be necessary. A minimal viable product for this feature branch could focus on demonstrating core value:

Connect Single Account: Support for connecting one email account (e.g., Gmail via API). Fetch & Store (Incoming Only): Reliably fetch incoming emails, parse essential headers and plaintext body, store as entry_type='email' with representations and linked assets for attachments. Read-Only View: Provide a basic UI within Onelist to list and read these fetched emails. Unified Search Integration: Ensure fetched emails are indexed and searchable via the core Onelist search. Manual Linking: Allow users to manually create entry_links between an email entry and another Onelist entry. (Defer: Sending/replying, multi-account, folder sync, complex daemon integration, specific workflow enhancements).

5. Future Evolution within this Branch

Full send/reply/forward capabilities. Robust multi-account support. Advanced folder/label mapping and management. Rich HTML email rendering (securely). Deep integration with Planner, Reader, and AI for task derivation, summarization, reply assistance, etc. Email rules and automation within Onelist. Offline support improvements.

6. Risks & Challenges

 Technical Complexity: Email protocols and APIs are complex and require significant effort to implement reliably. Synchronization state management is difficult. Security: Handling email credentials and rendering potentially unsafe email content securely are major challenges. OAuth2 implementations are required. Scalability: Potential for high volume of data (emails and attachments) requires robust storage and processing infrastructure. User Experience: Designing an integrated experience that is genuinely better than using a dedicated email client alongside Onelist, avoiding feature bloat. Third-Party Dependencies: Reliance on external email provider APIs and protocols introduces external risks (changes, rate limits, downtime). Resource Intensive: This is a very large feature requiring significant development effort across the stack.

Conclusion

Integrating email represents a significant potential expansion of Onelist's scope, strongly aligning with the vision of a unified "augmented memory." While technically challenging, successfully incorporating email would allow users to manage, search, link, and act upon a critical stream of personal and professional information directly within their central knowledge base, unlocking powerful new workflows and insights through Onelist's existing AI and daemon ecosystem. This direction would need careful consideration regarding its complexity and resource requirements relative to other potential post-v2.0 roadmap items.
